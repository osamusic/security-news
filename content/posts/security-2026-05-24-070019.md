### [Drupal Core SQL Injection Bug Actively Exploited, Added to CISA KEV](https://thehackernews.com/2026/05/drupal-core-sql-injection-bug-actively.html)
**2026-05-23 16:23 JST ／ The Hacker News ／ MEDIUM**


**要約:** Drupal Core のデータベース抽象化 API に存在する SQL インジェクション脆弱性 CVE-2026-9082（CVSS 6.5）が実環境で活発に悪用されており、CISA は KEV カタログに追加した。Imperva によると、65 か国 6,000 サイトに対して 15,000 件以上の攻撃試行が観測されており、特にゲームおよび金融分野が標的となっている。Drupal 11.3 / 11.2 / 11.1 / 10.6 / 10.5 / 10.4 向けにパッチが提供済みで、2026 年 5 月 27 日までの適用が推奨されている。


**なぜ重要か:** 悪用が既に確認された Tier-1 CMS の重大欠陥であり、リモートコード実行や権限昇格につながる。広く採用されている Drupal を直撃するため、攻撃面は世界的に大きい。


**影響:** Drupal を運用する Web アプリケーションでデータ抽出・改ざんや不正アクセスのリスクが急増する。ゲーム・金融セクターを中心に量的な攻撃キャンペーンが進行中。


**次のアクション:** 該当バージョンの Drupal を 5 月 27 日までに最新パッチへ更新する。WAF / アクセスログでデータベース抽象化 API に対する異常な POST/GET パラメータを監視し、KEV 期限内の対応を経営層に報告する。

### [LiteSpeed cPanel Plugin CVE-2026-48172 Exploited to Run Scripts as Root](https://thehackernews.com/2026/05/litespeed-cpanel-plugin-cve-2026-48172.html)
**2026-05-23 16:35 JST ／ The Hacker News ／ LOW**


**要約:** LiteSpeed User-End cPanel Plugin に CVSS 10.0 の最高深刻度の脆弱性 CVE-2026-48172 が発見され、実環境で悪用されている。権限割り当てミスにより、任意の cPanel ユーザ（攻撃者や侵害アカウントを含む）が昇格された権限で任意スクリプトを実行できる。


**要約:** ホスティング基盤の cPanel ユーザはマルチテナント運用の標準的な低権限アカウントであり、これが root 相当の実行権限を取得できることは事実上のフルテイクオーバーを意味する。


**なぜ重要か:** 共用ホスティング業者・VPS プロバイダで広く採用されている LiteSpeed プラグインを直撃し、CVSS 10.0 の root 権限奪取が可能。1 件の侵害が同一サーバ上の全テナントへ波及する。


**影響:** 多数のホスティング業者のサーバが横展開リスクにさらされる。テナント毎の侵害から root 化、永続化、追加のサプライチェーン攻撃へとつながる懸念がある。


**次のアクション:** LiteSpeed User-End cPanel Plugin を直ちにベンダ提供の修正版へ更新する。/usr/local/cpanel 配下や crontab、SUID ファイルの異常スクリプト混入を確認し、不審な cPanel アカウント作成・パスワード変更ログを精査する。

### ['Underminr' Vulnerability Lets Attackers Hide Malicious Connections Behind Trusted Domains](https://www.securityweek.com/underminr-vulnerability-lets-attackers-hide-malicious-connections-behind-trusted-domains/)
**2026-05-23 20:00 JST ／ SecurityWeek ／ LOW**


**要約:** 約 8,800 万ドメインに影響する「Underminr」と呼ばれるステルス脆弱性が発見された。本脆弱性を悪用すると、攻撃者は DNS フィルタリングを回避し、信頼されたドメインの裏に C2 通信を隠蔽できる。


**なぜ重要か:** DNS ベースのセキュリティ製品（DNS RPZ、セキュア DNS、EDR の DNS 監視）を素通りする経路を生む。脅威ハンティングや C2 検知の前提を揺るがす点で防御側への影響が大きい。


**影響:** DNS シンクホールやドメインブロックリストに依存した検知ロジックの実効性低下。マルウェア通信や情報窃取トラフィックが正規ドメインに紛れて見逃される可能性。


**次のアクション:** 自社ドメインが Underminr の影響対象に含まれるか確認し、DNS プロバイダの対策状況を照会する。Proxy・SSL インスペクションでドメイン外の SNI / Host ヘッダ不整合や TLS 証明書不一致を監視するルールを追加する。

### [Packagist Supply Chain Attack Infects 8 Packages Using GitHub-Hosted Linux Malware](https://thehackernews.com/2026/05/packagist-supply-chain-attack-infects-8.html)
**2026-05-24 01:07 JST ／ The Hacker News ／ MEDIUM**


**要約:** 協調的なサプライチェーン攻撃により、devdojo/wave、katanaui/katana を含む 8 つの Packagist パッケージが侵害された。攻撃者は PHP 向けセキュリティスキャンを回避するため、composer.json ではなく package.json に悪性コードを挿入し、postinstall スクリプトで GitHub から Linux バイナリ「gvfsd-network」をダウンロードして /tmp/.sshd としてバックグラウンド実行させる。同マルウェアは RCE 機能を備え、TLS 検証を無効化して活動を隠蔽する。GitHub 上の 700 ファイル以上で同ペイロードが参照されており、JavaScript ビルドツールを介して PHP プロジェクトを広範に狙うキャンペーンと判明している。


**なぜ重要か:** PHP エコシステム向けのスキャンを意図的に迂回した複合言語サプライチェーン攻撃であり、検知の盲点を突いている。Packagist の利用規模を考えると影響は広範。


**影響:** Composer 経由で侵害パッケージを導入したサーバが RCE 配下に置かれる。CI/CD 環境やビルドサーバでの自動展開を通じてさらに波及する恐れがある。


**次のアクション:** Composer.lock を確認し、対象 8 パッケージの侵害バージョンを排除する。/tmp/.sshd プロセスや gvfsd-network バイナリの存在、未知の postinstall 実行を EDR で検索する。PHP プロジェクトでも package.json をスキャン対象に含めるよう SCA / SBOM ツールを見直す。

### [npm Adds 2FA-Gated Publishing and Package Install Controls Against Supply Chain Attacks](https://thehackernews.com/2026/05/npm-adds-2fa-gated-publishing-and.html)
**2026-05-24 01:35 JST ／ The Hacker News ／ MEDIUM**


**要約:** GitHub が npm にサプライチェーン攻撃対策として「staged publishing」を導入。人間のメンテナが 2FA チャレンジを通過しないとパッケージが公開されない仕組みで、自動化された CI/CD ワークフローに対しても「存在証明（proof of presence）」を保証する。`npm stage publish`（CLI 11.15.0+）で利用可能。また `--allow-file` `--allow-remote` `--allow-directory` フラグを追加し、レジストリ外からのインストール元を制御できる。TeamPCP のような大規模毒入りグループからエコシステムを保護し、侵害された CI/CD シークレットからの不正リリースを防止する狙い。


**なぜ重要か:** Packagist や Laravel-Lang 侵害と同種のサプライチェーン攻撃が連続するなか、レジストリ側で根本対策が始まった意義は大きい。CI/CD トークン窃取後の自動公開を遮断できる。


**影響:** Node.js / フロントエンド開発のリリースフローに 2FA フローが組み込まれることになり、自動公開パイプラインの再設計が必要。一方で侵害トークン経由の悪意あるリリースは確実に減少する。


**次のアクション:** npm CLI を 11.15.0 以上にアップグレードし、社内パッケージ公開ワークフローを staged publishing 対応に移行する。`--allow-*` フラグの活用と、CI/CD の npm トークン権限を最小化するレビューを実施する。
